#### 5. 谈谈你对值类型和引用类型的理解？由此引发的装箱和拆箱是什么？在Unity中为什么要尽量避免装箱拆箱？
**【面试官考察意图】**：这是C#内存管理的绝对核心。不懂这个的客户端写出来的游戏会卡顿（GC问题）。
**【高分回答思路】**：
 * **内存位置**：值类型（如 int, float, struct, enum）通常分配在**线程的栈（Stack）**上，由系统自动管理，速度快；引用类型（如 class, string, 数组, 委托）分配在**托管堆（Heap）**上，栈上只保存一个指向堆内存的引用地址。
 * **装箱拆箱**：装箱就是把值类型转换为引用类型（比如把 int 赋值给 object），这会在堆上重新分配内存并复制数据；拆箱就是把引用类型转换回值类型。
 * **为什么避免**：在Unity中，**装箱操作会在托管堆上产生内存垃圾（Garbage）**。当堆内存不足时，会触发C#的GC（垃圾回收机制）。GC执行时会造成主线程卡顿（CPU峰值），在游戏里表现为明显的掉帧。所以我们在写 Debug.Log(字符串 + 数字) 或者频繁将结构体存入 ArrayList 等非泛型集合时，要格外小心。简历里我用了泛型集合和强类型避免了这些问题。

**【基础回答】**：
写一个简单的懒汉式单例：
```csharp
public class GameManager {
    private static GameManager instance;
    public static GameManager Instance {
        get {
            if (instance == null) instance = new GameManager();
            return instance;
        }
    }

```
**【高分进阶回答】**：指出上面的代码不是线程安全的，也不是Unity中继承 MonoBehaviour 的标准写法。
“在普通C#中，实现线程安全且高效的单例通常用双重检查锁定（Double-Check Locking）或者利用C#特性直接 private static readonly 初始化。
但在Unity中，如果单例继承自 MonoBehaviour，我们通常这样写来保证健壮性：”
```csharp
public class SoundManager : MonoBehaviour {
    public static SoundManager Instance { get; private set; }

    private void Awake() {
        if (Instance != null && Instance != this) {
            Destroy(this.gameObject); // 防止场景中有多个实例
            return;
        }
        Instance = this;
        DontDestroyOnLoad(this.gameObject); // 跨场景保留
    }
}

```
“我在项目中就是用这种方式管理全局音效和UI事件的。”

#### 8. C#中的委托（Delegate）和事件（Event）有什么区别？
**【面试官考察意图】**：简历中提到了“事件驱动模型，通过C#委托与事件实现...”。这是你的核心卖点。
**【高分回答思路】**：
“委托是一种**引用类型**，它类似于C++里的函数指针，可以存储对一个或多个具有相同签名的方法的引用。
事件（Event）则是**基于委托的一种特殊包装器（语法糖）**。
它们的核心区别在于**权限控制**：
如果我公开一个委托变量 public Action OnScoreChanged;，外部类不仅可以注册/注销方法（+=, -=），还可以直接在外部调用 OnScoreChanged?.Invoke()，甚至可以用 = 直接覆盖掉所有之前注册的方法，这破坏了封装性。
而使用 public event Action OnScoreChanged;，在类的外部**只能进行注册（+=）和注销（-=）操作，无法直接触发（Invoke）也无法直接赋值（=）**。
在项目中，我严守这个规范：只在逻辑核心层（比如计分器）定义 event 并触发，UI层只负责订阅事件并更新界面，彻底实现了数据和UI的解耦。”

#### 10. 简历中提到熟练使用 ScriptableObject。请问它的核心作用是什么？相比于 JSON 或 XML，它做配置有什么优势？
**【面试官考察意图】**：这是Unity特有的数据驱动方式。
**【高分回答思路】**：
“ScriptableObject (SO) 是一种独立于脚本实例之外的序列化数据容器。在《Kitchen Chaos》中，我用它来存储**菜谱配方**（比如汉堡需要什么切好的食材，煎烤需要多长时间等）。
**优势分析：**
 1. **内存优化**：如果是普通的 MonoBehaviour 存数据，当场景里有100个相同的汉堡时，这100个实例会各自在内存中复制一份数据。而使用 SO，由于它作为一种资源（Asset）存在，100个汉堡只需在内存中**引用同一份 SO 数据**，极大节省了内存。
 2. **策划友好**：SO 可以直接在 Unity 的 Project 窗口中右键创建（借助 [CreateAssetMenu] 属性），策划可以在 Inspector 面板直观地调整数值，甚至拖拽食材图标，不需要去修改容易格式出错的 JSON/XML 文件。
 3. **运行时修改保留（Editor下）**：在编辑器运行时修改 SO 的值，退出播放后修改是保留的，方便实时调参；但在真实打包后，它变成只读状态（除非特殊处理），保证了数据的安全性。”

**【基础回答】**：
熟练使用 Canvas, Text, Image, Button 等组件。知道锚点（Anchor）和屏幕适配。
**【高分进阶回答】**：
“我了解 UGUI 的渲染机制，它的核心痛点在于**网格重建（Rebuild）和 Draw Call 数量**。
因此在项目中我注意了以下几点优化：
 1. **动静分离**：Canvas 重建是整个 Canvas 级别触发的。如果把一个频繁更新的倒计时文本（动）和几十个不会变的背景图片（静）放在同一个 Canvas 下，文本的每次跳动都会导致所有图片重新计算网格。所以我把动态UI和静态UI拆分到了不同的 Canvas。
 2. **减少 Overdraw**：避免大量透明图片的层叠。对于不需要点击响应的背景图片，我会取消勾选 Raycast Target，以减少事件系统的射线检测开销。
 3. **图集（Sprite Atlas）压缩 Draw Call**：零散的 UI 图片如果不打包图集，每一张都会产生一个 Draw Call。我会利用图集工具把同一界面的小图打进一张大图里，只要它们的材质相同、不被打断，就能合并批次（Batching）。
 4. **对象池（Object Pool）**：对于频繁生成的UI元素（比如飘字伤害、弹幕、动态生成的食谱列表），绝对不要频繁 Instantiate 和 Destroy，而是使用对象池提前生成并做显隐控制。”

#### 13. 在《Kitchen Chaos》中，玩家从柜台上拿起（Pickup）物品和放下（Drop）物品的逻辑，你是如何保证严谨不穿模、不重影的？
**【面试官考察意图】**：考察业务逻辑的严密性。简历中写了“保证代码规范性与严谨逻辑”。
**【高分回答思路】**：
“这个逻辑是整个游戏的核心，涉及到物品所有权的转移。
我的设计思路是引入一个 IKitchenObjectParent 接口。玩家（Player）和各个台子（Counter）都实现了这个接口。接口规定了以下几个核心方法：

**交互的具体流程是这样的：**
当玩家对切菜板按下交互键：
首先，判断**玩家手里有没有东西（HasKitchenObject）**，同时判断**切菜板上有没有东西**。
 * **情况 A（玩家空手，台子上有番茄）**：玩家调用台子的 GetKitchenObject() 获取番茄的引用，接着调用台子的 ClearKitchenObject() 将台子清空。然后玩家调用自己的 SetKitchenObject() 成为番茄的新父级。最后，番茄脚本自身更新其位置，通过 Lerp 或者直接设置 localPosition 归零，贴合到玩家的持物挂载点上。
 * **情况 B（玩家手拿番茄，台子空）**：执行与上面完全相反的逻辑。玩家释放番茄，台子接收番茄。
通过这种**面向接口的方法调用，严格保证了任何时刻一个食材只能有唯一的一个父级（非黑即白）**，绝对不会出现玩家和台子同时持有同一个物体的Bug。”
#### 14. 如果现在要增加一个“可以切的番茄，切3刀变成番茄片，再切3刀变成番茄酱”的功能，你现有的架构需要做哪些修改？
**【面试官考察意图】**：考察代码的可扩展性（开闭原则）。
**【高分回答思路】**：
“由于我使用了 ScriptableObject（SO）来配置配方数据，扩展这个功能非常容易，并且**完全不需要修改核心切割逻辑的代码**。
现有的切割配方 SO (CuttingRecipeSO) 包含三个变量：输入物品 (番茄)、输出物品 (番茄片)、需要切的次数 (3)。
为了实现连切，我只需要在资源目录里：
 1. 配置一个 SO_1：Input=番茄，Output=番茄片，MaxCutCount=3。
 2. 配置一个 SO_2：Input=番茄片，Output=番茄酱，MaxCutCount=3。
   当玩家在切菜板上切东西时，切菜板的代码只需遍历所有的 CuttingRecipeSO，找到匹配当前放置物品的配方。切完3刀后，将“番茄”销毁，生成“番茄片”。
   如果玩家继续按切割键，代码会再次查找匹配“番茄片”的配方，找到 SO_2，然后继续累计切割次数，直到变成“番茄酱”。
   所以，**逻辑层不需要加任何 if-else 判断，完全依赖策划配置数据的驱动**。”
