# 二、第一版真正应该保留的核心系统

建议你把第一版压缩成 **5 个核心系统 + 1 个成长闭环**。

## 第一版必须保留的 5 个核心系统

### 1. 登录与选主公系统
保留原因：
- 这是玩家身份初始化入口
- 决定阵营和主城
- 决定后续数据存档基础结构

第一版做法：
- 新玩家首次进入：三选一主公
- 选择后写入数据：阵营、主公ID、等级=1、经验=0
- 老玩家直接读取存档
- 传送到对应主城出生点

---

### 2. 主城系统
保留原因：
- 它是所有玩法的中枢
- 也是未来扩展新系统的挂载点

第一版做法：
主城内只保留最基础功能点：
- 出生点
- 商店入口
- 升级入口
- 副本传送点
- 资源区传送点

注意：
第一版甚至可以先只做 **一个主城模板复制三份**，外观不同即可，功能完全一样。

---

### 3. 资源采集系统
保留原因：
- 它是最简单的“主动获取资源”玩法
- 可以构成养成消耗来源
- 也是服务端权威、防作弊、交互逻辑的良好练习

第一版做法：
- 资源区里只有两种资源点：
  - 树木 → 木材
  - 矿石 → 铁矿/矿石
- 玩家靠近并点击采集
- 读条 2~3 秒
- 服务端校验距离、资源状态、玩家状态
- 成功后发资源
- 节点进入冷却并刷新

---

### 4. 单人副本系统
保留原因：
- 它提供“挑战-奖励”核心乐趣
- 是战斗系统的承载容器
- 也是你练习实例化、状态管理的关键模块

第一版做法：
- 玩家在主殿进入副本
- 服务端克隆一个副本模板到远处隐藏坐标
- 玩家被传送进去
- 副本只有几组怪或一个Boss
- 通关后发奖励并销毁副本
- 失败/退出/掉线也销毁副本

---

### 5. 角色成长系统
保留原因：
- 没有成长，玩家就没有重复游玩动力
- 闭环的“反馈感”来自成长

第一版做法：
不要上太多养成线，先只保留 **一种主成长 + 一种附加成长**：

#### 主成长：主公等级
- 副本/采集获得经验
- 主公升级
- 主公等级提升玩家基础战力或解锁更高副本

#### 附加成长：装备强化 或 基础属性升级
你有两种简化路线可选：

##### 路线A：装备版
- 副本掉基础装备
- 装备可穿戴
- 木材/矿石用于升级装备
- 装备加属性

##### 路线B：主公直升版
- 不做复杂装备背包
- 玩家直接用资源升级“攻击 / 生命 / 防御”基础属性
- 实现更简单，但后续扩展不如装备版自然

如果站在“后期方便迭代”的角度，**我建议你第一版做路线A：基础装备系统**。
因为后面你可以自然扩展出：

- 装备品质
- 装备槽位
- 装备强化
- 套装
- 宝石

而不会推翻原架构。

# 《Roblox 三国养成游戏》简化版开发目标描述（MVP）

我需要制作一款 Roblox 游戏，参考三国养成类游戏的结构，但由于我是 Roblox 初学者，第一阶段只实现一个 **简化版、可玩闭环的 MVP**，重点是：

- 代码结构清晰
- 服务端权威
- 数据可持久化
- 架构方便后续逐步迭代

游戏第一版不追求还原复杂商业化养成系统，而是优先完成“玩家能进入游戏、完成基础成长循环并持续游玩”的最小可行版本。

---

## 一、第一版核心目标

玩家进入游戏后，能够完成以下完整流程：

1. 首次登录选择主公
2. 被传送到对应阵营主城
3. 在主城中前往资源区采集基础材料
4. 在主城中进入单人副本挑战怪物
5. 通过采集和副本获得资源、装备、经验
6. 通过升级主公和强化装备提升战力
7. 再次挑战更高收益内容

形成一个基础闭环：

> **选主公 → 主城 → 采集资源 / 打副本 → 获得经验和装备 → 升级自己 → 挑战更高内容**

---

## 二、第一版保留系统

### 1. 登录与选主公
- 新玩家首次进入时，弹出三选一主公界面
- 主公选择决定玩家所属阵营（魏 / 蜀 / 吴）
- 选择结果写入数据库
- 老玩家登录时直接读取存档，不再重复选择
- 登录后玩家被传送到所属主城出生点

### 2. 三个主城
- 魏、蜀、吴三座主城位于同一张地图不同超远坐标
- 玩家只能看到自己所在的主城区域
- 每个主城具备相同功能：
  - 出生区
  - 资源区传送点
  - 副本传送点
  - 商店 / 强化入口（可先用 UI 替代实体 NPC）

### 3. 公共资源区
- 所有阵营玩家共享一个资源区
- 资源区包含树木和矿石两种资源点
- 玩家可进行基础采集：
  - 树木产出木材
  - 矿石产出矿石
- 采集逻辑由服务端校验与发奖
- 资源点采集后自动刷新
- 第一版不开放 PvP

### 4. 单人副本
- 玩家通过主城传送点进入单人副本
- 服务端动态创建专属副本实例
- 玩家在副本中与怪物战斗
- 通关后获得经验、金币、装备或强化材料
- 副本结束后销毁实例并将玩家送回主城

### 5. 基础成长系统
第一版成长系统尽量简化，只保留以下内容：

#### 主公等级
- 玩家通过副本和部分玩法获得经验
- 主公升级后提升基础战斗能力，或作为装备强化/副本解锁条件

#### 装备系统
- 玩家可从副本或商店获得基础装备
- 装备可穿戴到角色身上
- 装备可以消耗资源进行强化
- 装备强化材料主要来自资源区采集

---

## 三、第一版不做的系统
以下系统全部进入未来迭代，不在第一版实现范围内：

- 武将抽卡
- 武将碎片与升星
- 武将技能系统
- 羁绊系统
- 战马系统
- 部曲系统
- 科技研发
- 内政经营
- 建筑升级
- 宝石系统
- 套装系统
- 主公技能系统
- 官职系统
- PvP 系统

## 四、第一版战斗设计
为降低复杂度，第一版副本战斗采用 **简化版自动回合制**：

- 玩家队伍可先简化为“玩家自身 + 预设战斗属性”
- 或者固定带 3 名初始武将（推荐后续再做）
- 怪物由副本配置生成
- 双方按照速度或固定顺序轮流攻击
- 每回合只进行普通攻击
- 伤害公式简化
- 战斗结果由服务端计算
- 客户端只负责战斗表现

如果为了更快落地，**第一版甚至可以先不做真实武将系统**，而是：

> 玩家当前战斗力 = 主公基础属性 + 装备加成

这样会比一开始就做 3 武将系统简单非常多。

---

## 五、第一版经济闭环
第一版经济体系只保留最基础资源：

- 金币：副本奖励，用于商店购买基础装备
- 木材：资源区树木采集获得，用于装备强化
- 矿石：资源区矿点采集获得，用于装备强化
- 装备：副本掉落或商店购买
- 经验：副本通关奖励，用于提升主公等级

形成闭环：

- 采集 → 获取木材/矿石
- 副本 → 获取金币/经验/装备
- 强化 → 消耗木材/矿石/金币
- 升级 → 提升战力
- 更强战力 → 打更高收益副本

---

## 六、第一版架构要求

### 1. 单地图架构
- 所有场景位于同一个 Place
- 主城、资源区、副本实例通过远坐标隔离
- 场景切换统一使用 `PivotTo`

### 2. 服务端权威
以下逻辑必须在 Server 处理：
- 玩家数据初始化与保存
- 主公选择确认
- 传送合法性
- 资源采集判定
- 奖励发放
- 商店扣费
- 装备强化
- 副本实例管理
- 战斗结算

### 3. 数据持久化
- 使用 ProfileService
- 玩家离线自动保存
- 定时自动保存
- 所有关键成长数据持久化

### 4. 配置驱动
以下内容应尽量做成配置表：
- 主城坐标
- 阵营出生点
- 资源点奖励
- 副本怪物属性
- 副本奖励
- 装备属性
- 装备强化消耗
- 主公升级经验表

---

## 八、第一版开发原则
由于我是 Roblox 初学者，后续开发文档必须遵循：

1. 先做最小可运行版本，不追求复杂系统
2. 每个功能必须按步骤实现，不能跳步骤
3. 优先实现服务端逻辑，再补客户端 UI
4. 所有系统要为后续扩展留接口
5. 如果某个系统后续会扩展，第一版先做最简版本，但代码结构不要写死
