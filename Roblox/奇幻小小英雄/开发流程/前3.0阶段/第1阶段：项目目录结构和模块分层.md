下面开始 **第 1 阶段：项目目录结构和模块分层**。
目标只有一个：

> **先把 Roblox Studio 里的工程骨架搭好，保证后面每个系统都有明确归属，不乱写。**

这一阶段我们先不做复杂逻辑，重点是：

- 目录怎么建
- 每个脚本放哪里
- Server / Client / Shared 怎么分
- Remote 怎么管理
- Workspace 场景对象怎么命名
- 为什么这样设计更适合后续迭代

我会尽量按 **你现在就能在 Roblox Studio 里一步一步照着操作** 的方式来写。

# 一、第 1 阶段的目标
---

当你完成这一阶段后，你的项目应该具备这些基础条件：

- 有清晰的文件夹结构
- 有统一的模块分层
- 有共享配置、共享常量、共享工具的放置位置
- 有服务端系统脚本的放置位置
- 有客户端 UI / 交互脚本的放置位置
- 有 RemoteEvents / RemoteFunctions 的统一管理位置
- 有地图区域的标准命名方式
- 后面接数据系统、选主公系统时不会返工

# 二、先理解 Roblox 的三层分工
---

在开始建目录之前，你先必须理解一个最基础的架构思想：

## 1. Server 层
放在服务器运行的代码。

主要职责：
- 数据加载和保存
- 校验玩家请求
- 处理奖励
- 控制传送
- 副本实例管理
- 战斗计算
- 防作弊

常见放置位置：
- `ServerScriptService`
- 有些服务端资源对象可能放在 `ServerStorage`

---

## 2. Client 层
放在玩家客户端运行的代码。

主要职责：
- UI显示
- 动画
- 特效
- 按钮点击
- 发送请求给服务端
- 本地交互反馈

常见放置位置：
- `StarterPlayer > StarterPlayerScripts`
- `StarterGui`

---

## 3. Shared 层
客户端和服务端都可能会用到的代码和配置。

主要职责：
- 常量定义
- 配置表
- 公共类型/枚举
- 纯逻辑工具函数
- Remote 名称表

常见放置位置：
- `ReplicatedStorage`

---

## 4. 为什么要这么分
因为 Roblox 项目最容易犯的错就是：

- 什么都丢进一个 Script
- 客户端偷偷算奖励
- 服务端和客户端都各写一套配置
- Remote 随手创建、命名混乱
- 后期完全没法维护

你现在目标是“成为 Roblox 全栈开发”，所以从第一天开始就要养成工程化习惯。

2. 右键 `ReplicatedStorage`
3. 选择 `Insert Object`
4. 创建 `Folder`
5. 重命名为：

```text
Shared
```

再创建另一个 `Folder`，命名为：

```text

```

---

## Step 1.2：在 Shared 下创建 4 个文件夹

右键 `Shared`，依次创建这些 `Folder`：

```text
Config
Constants
Types
Utils
```

最终结果：

```text
ReplicatedStorage
└─ Shared
   ├─ Config
   ├─ Constants
   ├─ Types
   └─ Utils
```

---

## Step 1.3：在 Remotes 下创建 2 个文件夹

右键 `Remotes`，创建：

```text
Events
Functions
```

最终结果：

```text
ReplicatedStorage
└─ Remotes
   ├─ Events
   └─ Functions
```

---

## 为什么这样分

### Config
放配置表，例如：
- 阵营出生点配置
- 副本奖励配置
- 资源节点配置
- 升级经验表

### Constants
放常量，例如：
- 阵营名字
- Remote 名字
- 地图区域名
- 装备槽位名

### Types
如果你后面想做更规范的数据类型定义，可以放这里。
第一版可以先留空。

### Utils
放纯工具函数，例如：
- table 深拷贝
- 数字格式化
- 安全查找函数

### Remotes
统一存网络通信对象，避免散落。

# 六、Step 2：先建立 ServerScriptService 结构
---

这里是服务端核心逻辑区。

---

## Step 2.1：在 ServerScriptService 下创建顶层 Folder

2. 插入一个 `Folder`
3. 命名为：

```text

```

## Step 2.2：在 Server 下创建 4 个文件夹

依次创建：

```text

Services

```

最终结果：

```text
ServerScriptService
└─ Server
   ├─ Bootstrap
   ├─ Services
   ├─ Systems
   └─ Managers
```

---

## 为什么要这样分

这个很重要。

### Bootstrap
放启动脚本。
作用是：服务器启动时，把各个系统初始化起来。

比如以后你会有：
- DataService 启动
- PlayerService 启动
- ResourceSystem 启动

---

### Services
放“全局服务类模块”。

特点：
- 生命周期长
- 为多个系统提供能力
- 通常管理一类核心领域

比如：
- `DataService`
- `PlayerService`
- `TeleportService`
- `DungeonService`

---

### Systems
放具体玩法系统。

特点：
- 更接近某个功能玩法
- 使用 Services 提供的数据和能力

比如：
- `SelectionSystem`
- `ResourceSystem`
- `BattleSystem`
- `ShopSystem`

---

### Managers
放资源管理器、对象注册器、状态管理器。

比如：
- `RemoteManager`
- `ResourceNodeManager`
- `DungeonInstanceManager`

---

## 新手阶段要不要这么细？
要。
因为你的目标不是做一个脚本堆，而是做一个**可维护项目**。

但第一版你不用一次塞满内容，先把位置留好。

# 十二、Step 8：为出生点预留结构
---

在 `Workspace > Map > SpawnPoints` 下创建 3 个 `Part` 作为临时出生点：

```text

```

### 操作方式
1. 右键 `SpawnPoints`
2. 插入 `Part`
3. 重命名
4. 将 `Anchored` 设为 `true`
5. 将 `CanCollide` 设为 `false`
6. 适当调整颜色方便识别

你可以先把它们摆在不同位置，后面再移动到主城实际位置。

## 为什么用 Part 而不是 SpawnLocation
因为你后面会自己控制玩家出生与传送逻辑，
用普通 `Part` 作为“标记点”更灵活。

# 二十一、Step 17：创建 Remote 对象容器
---

现在先不创建一堆 Remote，
但至少要在 `ReplicatedStorage > Remotes` 下准备规则。

建议先在 `Events` 下创建这些 `RemoteEvent`：

```text

ResourceUpdate
TeleportRequest
DungeonRequest
```

在 `Functions` 下先可以留空，或者后面需要同步请求时再加。

---

## 但更推荐的做法
你可以先**只建文件夹，不手动建 RemoteEvent**，
后面让 `RemoteManager` 在服务器启动时自动创建。

为什么我更推荐自动创建？

- 避免手工漏建
- 命名统一
- 后期更规范

所以目前阶段你可以先不建具体 Event，只保留：
- `Events`
- `Functions`
