
## 1. 目录结构：严守“职能分离”原则

Roblox 的游戏资源是按照特定的 Service（服务）分类的。不要把逻辑脚本混在 Workspace 的零件里。

|**目录 (Service)**|**存放内容**|**作用**|
|---|---|---|
|**Workspace**|仅存放具有物理实体的零件、模型。|玩家能看见和碰撞的世界。|
|**ReplicatedStorage**|远程事件 (RemoteEvents)、模块脚本 (ModuleScripts)、共享配置。|**客户端和服务器都能访问**，全栈通信的核心。|
|**ServerScriptService**|服务器端逻辑脚本 (Script)。|只有服务器能看，放数据库读写、伤害判定等敏感逻辑。|
|**StarterPlayerScripts**|客户端逻辑 (LocalScript)。|只有玩家本地运行，放 UI 交互、本地特效、输入检测。|
|**ServerStorage**|备用模型、地图预制件。|只有服务器可见，适合放还未生成的地图关卡或掉落物。|

## 2. 命名规范：告别 "Part" 和 "Script"

默认的命名是新手最大的敌人。

- **大驼峰命名 (PascalCase)：** Roblox 官方 API 使用 PascalCase（如 `Instance.new("Part")`），建议你的所有对象名、属性名、函数名也统一使用 `GameManager` 而不是 `game_manager`。
    
- **语义化命名：** 严禁出现 `Part1`, `Part2`, `Script`。
    
    - 坏例子：`Part`（在 Workspace 里有 100 个叫 Part 的东西）。
        
    - 好例子：`MainBase`, `LavaFloor`, `HealthPickup`。
        
- **前缀标识（可选）：** 有些开发者喜欢给脚本加前缀，例如 `Client_InputHandler` 或 `Server_DataStore`，方便在搜索栏快速定位。
    

---

## 3. 全栈开发的“模块化”思维

Roblox 全栈开发的核心是 **ModuleScript（模块脚本）**。不要在每个 Script 里重写一遍同样的功能。

- **Don't Repeat Yourself (DRY)：** 如果你有多个地方要用到“玩家金币增加”的逻辑，写一个 ModuleScript 放在 `ReplicatedStorage` 或 `ServerScriptService`。
    
- **单脚本架构 (Single Script Architecture)：** 进阶习惯是：
    
    - 服务器只运行一个入口 Script，然后去调用各个 Module。
        
    - 客户端只运行一个入口 LocalScript。
        
    - 这样可以极大地避免脚本冲突，并能精确控制初始化顺序。

## 4. Part（零件）管理的工程化

在 Workspace 里的零件，如果不规范，性能会迅速下降：

- **模型化 (Grouping)：** 相关的零件务必组合成 `Model`。例如一棵树，由 Trunk 和 Leaves 组成，应放入 `Tree` 模型中。
    
- **锚定 (Anchoring)：** 除非零件需要动（如小球、NPC），否则**全部勾选 Anchored**。这能减轻物理引擎（Physics Engine）的计算负担。
    
- **外观 vs 碰撞：** 复杂的装饰品可以关闭 `CanCollide` 和 `CanTouch`，甚至开启 `Massless`，优化性能。
    
- **标签化 (CollectionService)：** 如果你有 50 个岩浆块，不要在每个里面写脚本。给它们打上 "Lava" 标签，然后用一个脚本统一管理。

## 5. 现代开发流程：同步工具

如果你想更进一步，像真正的全栈工程师一样工作，建议尝试 **Rojo**。

- **Rojo：** 它可以让你在 **VS Code** 里写 Luau 代码，并自动同步到 Roblox Studio。
    
- **版本控制：** 这样你就可以使用 **Git/GitHub** 来管理你的代码版本，而不是依赖 Roblox 自带的那份极其脆弱的“云端保存”。
    

> **建议：** 从今天开始，每创建一个 `Part` 或 `Script`，强迫自己第一时间重命名，并思考它应该放在哪个 Service 里。如果你能坚持一周，你的项目结构会比 90% 的新手都要整洁。
