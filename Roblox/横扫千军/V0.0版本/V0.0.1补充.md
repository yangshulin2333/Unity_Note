## 玩家进入服务器时，系统发生了什么？

> 把这个角色传送到他自己的 Plot 出生点。

### 6. 服务端告诉客户端
服务端通过 Remote 告诉玩家自己的客户端：

> 你叫谁
> 你几级
> 你有多少钱
> 你分到了哪块地

客户端的 `UIController` 收到消息后，把文字显示在 HUD 上。

# 四、先彻底搞懂 Roblox 里最重要的三个脚本类型

如果这一块不懂，后面永远会懵。

---

## 1. Script

运行位置：**服务端**

服务端脚本负责：
- 核心规则
- 数据
- 判定
- 安全逻辑
- 多人同步

比如：
- 分 Plot
- 加载数据
- 传送玩家
- 扣钱
- 升级
- 掉落

第一阶段里的：
- `Main.server`
- `DataService`
- `PlayerService`
- `BaseService`

本质都属于服务端逻辑。

---

## 2. LocalScript

运行位置：**客户端**

客户端脚本负责：
- UI
- 本地表现
- 输入监听
- 镜头
- 特效
- 动画展示

第一阶段里的：
- `ClientMain`

它是在玩家自己的电脑上运行的。

---

## 3. ModuleScript

运行位置：**看谁 require 它**

ModuleScript 本身不是自动运行的。
它像一个“工具包”或者“功能模块”。

例如：
- `DataService`
- `PlayerService`
- `BaseService`
- `UIController`
- `PlotController`

这些都是模块。
谁 `require()` 它，它才参与工作。

# 五、你必须学会的第一原则：客户端不可信

这是所有网络游戏最关键的思想之一。

---

## 为什么客户端不可信？

因为客户端在玩家电脑上。
玩家可以：
- 修改本地脚本
- 注入作弊代码
- 伪造请求

所以客户端不能负责：
- 发钱
- 判定战斗胜负
- 决定升级成功
- 保存真实数据

---

## 客户端能做什么？

客户端可以：
- 显示 UI
- 接受按键输入
- 发起请求
- 播放动画
- 展示结果

---

## 第一阶段中的例子

### 服务端做的事
- 玩家数据初始化
- Plot 分配
- 角色传送

### 客户端做的事
- 显示名字
- 显示等级
- 显示铜币

这就是正确的职责划分。

# 六、逐个讲你第一阶段的目录，为什么这样建

现在开始真正“拆目录”。

---

## 1. ReplicatedStorage

这是一个**客户端和服务端都能访问**的地方。

适合放：
- 共享配置
- Remote
- 共享模块
- 公共资源

你第一阶段建的：

```text
ReplicatedStorage
├── Shared
│   ├── Config
│   ├── Constants
│   ├── Utils
│   ├── Types
│   └── Remotes
├── Assets
└── Units
```

---

### 为什么 `Remotes` 放这里？
因为 Remote 要同时被：
- 服务端找到
- 客户端找到

所以必须放到两边都能访问的地方。
`ReplicatedStorage` 最合适。

---

### 为什么 `Shared` 要单独建？
因为你以后会有很多“双方都能用的内容”，比如：
- 建筑配置
- 武将配置
- 常量
- 工具函数
- 类型定义

统一放在 `Shared`，不会乱。

---

## 2. ServerScriptService

这里放**服务端逻辑脚本**。

第一阶段你放了：
- `Services`
- `Systems`
- `Main.server`

---

### 为什么 `Main.server` 存在？
因为你需要一个“总入口”。

就像程序启动器一样。
它的工作不是写全部逻辑，
而是：

> 把各个服务加载起来，让系统开始运行。

所以：

```lua
local PlayerService = require(...)
PlayerService:Init()
```

这就是“启动 PlayerService”。

---

### 为什么要有 Services？
因为大型项目里，逻辑要按职责拆。

比如：
- `DataService` 专门管数据
- `PlayerService` 专门管玩家生命周期
- `BaseService` 专门管地块/基地逻辑

你不能把所有事情都写进一个 Script 里。

---

## 3. ServerStorage

这里放**只能服务端访问**的东西。

例如：
- 模板
- 默认配置
- 敌人模板
- 副本模板
- 默认数据模板

第一阶段放了：
- `Templates`
- `Data/DefaultProfile`

---

### 为什么 `DefaultProfile` 放这里？
因为玩家默认数据是服务端核心数据，
客户端不需要直接改它。

---

## 4. StarterPlayerScripts

这里放玩家进入游戏后，自动复制到客户端运行的脚本。

你第一阶段放了：
- `ClientMain`
- `Controllers`

---

### 为什么客户端也要做模块化？
因为客户端以后也会越来越复杂：

- UIController
- CameraController
- BattleController
- HeroController
- AudioController

如果一开始不拆，后面会炸。

---

## 5. StarterGui

这里放 UI。

第一阶段你有：
- `MainHUD`
- `LoadingScreen`

这很正常。

---

## 6. Workspace

这里是游戏世界本体。

你第一阶段放了：
- `Plots`
- `ActiveDungeons`
- `WorldMarkers`

其中第一阶段真正用到的是：
- `Plots`

# 七、第一阶段每个核心脚本到底负责什么

这是最关键的一部分。

---

## 1. DefaultProfile：玩家默认数据模板

它本质上是：

# **“新玩家应该长什么样”**

比如：

```lua
Currency = {
    Coins = 500,
    Food = 100,
    Iron = 50,
    Gems = 0,
}
```

表示新玩家初始资源。

---

### 为什么要有它？
因为每个玩家都要有一份初始数据。
你不能每次手写。

所以我们先定义一份标准模板，
玩家进来就复制一份。

---

### 你要记住一句话：
`DefaultProfile` 不是玩家数据本体。
它只是“模板”。

真正玩家的数据，是复制出来后放进 `DataService.Profiles[player]` 里的那份。

---

## 2. DataService：数据管理员

`DataService` 的职责是：

# **管理玩家数据**

第一阶段它做了三件事：

1. `LoadProfile(player)`
   给玩家创建一份数据

2. `GetProfile(player)`
   获取这名玩家当前数据

3. `SaveProfile(player)` / `ReleaseProfile(player)`
   保存并释放数据

---

### 它为什么叫 Service？
因为它提供的是“数据服务”。

其他脚本不用知道数据怎么创建、怎么存。
只需要说：

```lua
local profile = DataService:GetProfile(player)
```

这就是“向 DataService 要数据”。

---

### 第一阶段为什么不用正式 DataStore？
因为你现在在学架构，不是先学云存档。
先把调用结构跑通，再接正式存档，这是对的。

---

## 3. BaseService：基地/地块管理员

它负责：

# **谁用哪块 Plot**

以及和 Plot 相关的逻辑。

第一阶段它做了这些：

1. 获取所有 Plot
2. 检查 Plot 是否空闲
3. 给玩家分 Plot
4. 玩家离开时释放 Plot
5. 取 Plot 的出生点

---

### 为什么这部分不写进 PlayerService？
因为“玩家逻辑”和“Plot 逻辑”虽然相关，但不是一回事。

- 玩家逻辑：加入、离开、角色出生
- Plot 逻辑：地块查找、空闲检查、分配规则

拆开后更清楚。

---

## 4. PlayerService：玩家生命周期管理员

它是第一阶段最重要的“协调者”。

它负责：

1. 玩家加入
2. 玩家离开
3. 角色生成
4. 调 DataService 加载数据
5. 调 BaseService 分配地块
6. 把初始化数据发给客户端

---

### 你可以把它理解成：
# **玩家总接待台**

玩家一进来，先过它这关。

---

### 它和 DataService/BaseService 的关系
它自己不做所有底层工作，
而是协调别的 Service。

例如：

```lua
local profile = DataService:LoadProfile(player)
local plot = BaseService:AssignPlot(player)
```

这说明：
- 数据让数据服务负责
- 地块让基地服务负责
- PlayerService 只负责把流程串起来

这就是好架构。

---

## 5. Main.server：服务端启动器

它的职责极其简单：

```lua
PlayerService:Init()
```

就是告诉 PlayerService：

> 开始监听玩家加入和离开吧。

---

## 6. UIController：UI 管理器

客户端模块之一。

它负责：
- 找到 UI 控件
- 修改标题文本
- 显示玩家等级
- 显示玩家货币

---

### 为什么 UI 单独做成 Controller？
因为 UI 更新以后会很多。
如果全写在 `ClientMain` 里，后面一定乱。

---

## 7. PlotController：客户端 Plot 状态记录器

第一阶段它很简单，只负责：
- 记录当前被分配到了哪个 Plot

以后它可以扩展为：
- 高亮自己的地块
- 小地图显示归属
- 地块范围显示
- 建筑节点提示

所以现在虽然简单，但方向是对的。

---

## 8. ClientMain：客户端启动器

和 `Main.server` 一样，它是客户端总入口。

它做了什么？
1. require UIController
2. require PlotController
3. 初始化它们
4. 监听服务端发来的 Remote
5. 请求初始玩家数据

---

# 八、第一阶段最重要的数据流：你必须彻底搞懂

现在我们讲“玩家进入后数据怎么流动”。

这部分你必须搞懂，否则后面所有系统都会乱。

---

## 流程 1：玩家加入服务器

### 第一步
玩家加入游戏

Roblox 自动触发：
```lua
Players.PlayerAdded
```

---

### 第二步
`PlayerService:OnPlayerAdded(player)` 被调用

---

### 第三步
调用：
```lua
local profile = DataService:LoadProfile(player)
```

意思是：
- 创建玩家数据
- 存到 `DataService.Profiles[player]`

---

### 第四步
调用：
```lua
local plot = BaseService:AssignPlot(player)
```

意思是：
- 查找空闲 Plot
- 分配给玩家
- 记录到 `BaseService.AssignedPlots[player]`

---

### 第五步
绑定角色出生事件：

```lua
player.CharacterAdded:Connect(function(character)
    self:OnCharacterAdded(player, character)
end)
```

意思是：
> 等会儿这个玩家角色真正生成出来时，我要处理传送。

---

### 第六步
服务端发 Remote 给客户端：

```lua
initClientRemote:FireClient(player, {...})
```

意思是：
> 把初始化数据显示给这个玩家自己的客户端。

---

## 流程 2：玩家角色生成

当角色生成后：

```lua
PlayerService:OnCharacterAdded(player, character)
```

执行：

1. 找到该玩家分到的 Plot
2. 找到 Plot 下的 `BaseSpawn`
3. 找到角色的 `HumanoidRootPart`
4. 修改 `CFrame`
5. 把角色移动到 Plot 出生点附近

---

## 流程 3：客户端更新 UI

客户端监听：

```lua
initClientRemote.OnClientEvent:Connect(function(data)
    UIController:UpdateInitData(data)
end)
```

服务端一发消息，客户端立刻收到。
收到后 UIController 就改文字。

---

# 九、为什么要先 `FireClient`，还要 `RequestPlayerData`？

你可能会疑惑：
不是已经发了初始化数据吗？
为什么又有 `RequestPlayerData`？

这是个非常好的问题。

---

## 原因
这是在练两种通信方式。

---

### 1. `RemoteEvent`
服务端主动推送

例如：
```lua
initClientRemote:FireClient(player, data)
```

意思是：
> 我主动告诉你

---

### 2. `RemoteFunction`
客户端主动请求

例如：
```lua
requestPlayerDataFunction:InvokeServer()
```

意思是：
> 我来问你，你立刻回答我

---

### 为什么两种都要学？
因为以后你一定会同时用到：

#### 服务端主动推送
- 奖励到了
- 副本结束
- 资源变化
- 倒计时开始

#### 客户端主动请求
- 打开建筑面板
- 查询背包
- 查询英雄详情
- 查询排行

所以第一阶段同时接触两种方式是好事。

---

# 十、逐行讲你最该理解的几段代码

现在我挑最关键的几段，不是每行都讲，而是讲“你一定要懂的点”。

---

## 1. `DataService:LoadProfile(player)`

```lua
local profile = deepCopy(DefaultProfile)
profile.Player.Name = player.Name
profile.Timers.LastLogin = os.time()

self.Profiles[player] = profile
return profile
```

---

### 这段在干嘛？

#### `deepCopy(DefaultProfile)`
复制一份默认模板。
不能直接用原表，否则所有玩家会共用同一份数据，出大问题。

#### `profile.Player.Name = player.Name`
把这个玩家的真实名字写进去。

#### `self.Profiles[player] = profile`
把这份玩家数据记到 DataService 里，方便以后查询。

---

### 你必须明白：
这里的 `self.Profiles[player]` 才是“当前玩家数据”。

---

## 2. `BaseService:AssignPlot(player)`

```lua
for _, plot in ipairs(self:GetAllPlots()) do
    if self:IsPlotFree(plot) then
        self.AssignedPlots[player] = plot
        return plot
    end
end
```

---

### 这段在干嘛？
遍历所有 Plot，找到第一个空闲地块，分给该玩家。

---

### 为什么能记录玩家和 Plot 的关系？
因为这句：

```lua
self.AssignedPlots[player] = plot
```

本质是在做一张映射表：

| 玩家 | Plot |
|---|---|
| 玩家A | Plot_1 |
| 玩家B | Plot_2 |

---

## 3. `PlayerService:OnCharacterAdded(player, character)`

```lua
local hrp = character:WaitForChild("HumanoidRootPart", 5)
if hrp then
    hrp.CFrame = spawnPart.CFrame + Vector3.new(0, 3, 0)
end
```

---

### 这段在干嘛？
角色模型刚出生，不一定所有部件立刻都有。
所以先等 `HumanoidRootPart` 出现。

有了它后，直接改 `CFrame` 把角色移动到指定位置。

---

### 为什么加 `Vector3.new(0,3,0)`？
为了避免角色直接卡进地面。
往上抬一点更安全。

---

## 4. `ClientMain` 里监听 Remote

```lua
initClientRemote.OnClientEvent:Connect(function(data)
    UIController:UpdateInitData(data)
end)
```

---

### 这段是什么意思？
服务端发消息 -> 客户端收消息 -> UI 更新

这是多人游戏最基本的数据同步模式之一。

---

# 十一、你必须建立的“分层思维”

以后你每加一个功能，都要问自己：

---

## 这个功能属于哪一层？

### 1. 数据层
这个功能需要哪些数据？
比如武将系统需要：
- HeroId
- Level
- Star
- Skills

### 2. 规则层
这个功能规则是什么？
比如升级英雄：
- 要消耗经验丹
- 要消耗铜币
- 有等级上限

### 3. 表现层
这个功能怎么给玩家看？
- 英雄面板
- 升级按钮
- 数值变化

### 4. 通信层
客户端和服务端怎么交互？
- 点击升级 -> 发请求
- 服务端判断 -> 返回结果

---

如果你以后都能这么想，你就不会乱。

# 十二、你未来要改成“魏蜀吴三选一主城”，第一阶段架构还能用吗？
