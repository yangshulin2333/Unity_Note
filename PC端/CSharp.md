### 1. 面向对象三大特性：封装、继承、多态（必考！）

这是所有面向对象语言的基础，但在 Unity 中有非常具体的应用场景。

- **封装 (Encapsulation)：保护数据**
    
    - **怎么说：** “我习惯把字段设为 `private`，然后通过属性（Property）或者公开的方法暴露出去。比如 `public bool IsPlaying { get; private set; }`。”
        
    - **项目对应：** 游戏中任何状态（比如是否拿着东西），外部脚本只能“读取（get）”，不能直接“修改（set）”，修改必须通过特定的交互方法。这能防止代码乱套。
        
- **继承 (Inheritance)：复用代码**
    
    - **怎么说：** “为了减少冗余代码，我会把公用的逻辑提取到父类中。”
        
    - **项目对应：** 《Kitchen Chaos》里所有的桌台（空桌子、切菜板、炉子）都继承自一个基类 `BaseCounter`。`BaseCounter` 处理了桌台最基础的逻辑（比如桌子上有没有东西），子类直接继承，不用重写。
        
- **多态 (Polymorphism)：同一种指令，不同表现**
    
    - **怎么说：** “我通过 `virtual`（虚方法）和 `override`（重写）来实现多态。”
        
    - **项目对应：** `BaseCounter` 里有一个 `public virtual void Interact(Player player)` 方法。玩家按下交互键时，统一调用这个方法。但切菜板重写（`override`）了这个方法去执行切菜，炉子重写去执行点火。玩家脚本根本不需要知道面前是什么桌子，调用 `Interact` 就行了。
        

### 2. 常用数据结构：List 与 Dictionary

面试官很喜欢问：“你在项目中怎么存储数据的？为什么用这个不用那个？”

- **`List<T>` (列表)**
    
    - **特点：** 大小可变的数组，内存中是连续分配的（大部分情况下）。
        
    - **项目对应：** 订单系统。当前需要做的汉堡、沙拉，就是一个 `List<RecipeSO>`。每次完成一个订单，就从 List 里 `Remove` 掉；来新订单，就 `Add` 进来。
        
- **`Dictionary<TKey, TValue>` (字典)**
    
    - **特点：** 键值对存储，通过 Hash 算法实现，**查找速度极快（时间复杂度接近 O(1)）**。
        
    - **怎么说：** “如果需要频繁查找某个数据，我会优先考虑字典而不是去遍历 List。”
        
    - **面试防坑：** 虽然字典查找快，但在 Unity 中遍历字典会产生一点点 GC（垃圾回收），所以如果只是为了按顺序存取，用 List；如果是“通过 ID 找物品”，用 Dictionary。
        

### 3. C# 特有语法：Action、Func 与 EventHandler

你前面说到了事件和委托，CodeMonkey 在项目里大量使用了 C# 自带的委托类型。面试官可能会问：“你自己写过 delegate 吗？还是直接用系统自带的？”

- **`Action` 和 `Action<T>`：**
    
    - 这是 C# 封装好的委托。`Action` 不带参数，没有返回值。`Action<int>` 带一个 int 参数，没有返回值。
        
    - **应用：** 比如 UI 按钮的点击回调，或者简单的状态通知。
        
- **`EventHandler` 和 `EventHandler<TEventArgs>`（项目核心应用）：**
    
    - **怎么说：** “项目中我主要遵循微软的规范，使用 `EventHandler` 来处理事件。它的好处是默认带了 `object sender`（谁触发的事件）和 `EventArgs`（事件参数），非常标准。”
        
    - **项目对应：** 切菜板切完一刀，触发 `OnProgressChanged` 事件，把当前的切菜进度（一个 float 值）包装在 `EventArgs` 里传给 UI。
        

### 4. 值类型 (Value Type) vs 引用类型 (Reference Type)

这是 C# 面试的基础中的基础，几乎逢面必问！

- **值类型：**
    
    - 包括 `int`, `float`, `bool`, `struct`（结构体）, `enum`（枚举）。
        
    - **存在哪：** 存在**栈（Stack）**上。
        
    - **怎么传递：** 传递的是**数据的副本**（你给我一份复印件，我撕了对你的原件没影响）。
        
- **引用类型：**
    
    - 包括 `class`（类）, `string`, 数组, 委托。
        
    - **存在哪：** 数据存在**堆（Heap）**上，栈上只保存一个指向堆内存的内存地址（指针）。
        
    - **怎么传递：** 传递的是**内存地址**（给你我家钥匙，你进屋把电视砸了，我家的电视就真没了）。
        
- **装箱与拆箱 (Boxing & Unboxing)：** （如果他问深了就抛出这个词）把值类型转换成引用类型叫装箱，反之叫拆箱。**装箱会产生内存垃圾，游戏开发中要尽量避免。**
    

### 5. 垃圾回收机制 (GC - Garbage Collection)

面试官如果想看你的“游戏性能优化”意识，一定会问 GC。

- **什么是 GC？** C# 会自动帮你清理那些没用的“引用类型”内存（堆内存）。
    
- **为什么游戏开发怕 GC？** 清理垃圾时，会造成 CPU 性能消耗，导致游戏**瞬间卡顿（掉帧）**。
    
- **如何避免 GC？（面试高分回答）：**
    
    1. **绝对不要在 `Update()` 里面 `new` 新对象。** （比如 `new List()` 或者实例化类的操作），因为 Update 每秒执行 60 次，这会疯狂制造垃圾。
        
    2. **避免频繁的字符串拼接。** （比如 `scoreText.text = "Score: " + score;` 会产生 GC，在复杂项目中应该缓存字符串或使用 `StringBuilder`）。
        
    3. **使用对象池 (Object Pool)。**
        
        - **项目延伸：** 虽然《Kitchen Chaos》里可能没深究，但你可以主动说：“比如游戏里切菜时飞出来的菜叶粒子特效，如果不优化，频繁 Instantiate 和 Destroy 会产生大量 GC。更好的做法是写一个对象池，一开始生成 10 块菜叶隐藏起来，用的时候拿出来，用完再隐藏回收，重复利用。”老板听到这个会觉得你很有实战意识。
